# 代码重构方法论

本文档从软件设计本质层面，提炼可复用的重构方法论，指导代码质量改进。

---

## 一、设计哲学与原理

### 1.1 代码为什么会退化

**退化的根源**：知识分裂、快速修复、设计腐化

```
初始设计 → 业务增长 → 快速迭代 → 技术债务 → 代码退化
    ↓                              ↓
  一致性                         分裂
  抽象                           具体化
  意图清晰                       意图模糊
```

**退化机制**：
- **复制粘贴** → 知识分裂，修改一处忘另一处
- **快速修复** → 破坏原有设计意图
- **需求变更** → 原设计不再适用，补丁叠加
- **知识遗忘** → 不知道为什么要这样写

**退化规律**：所有代码都会退化，重构是必然的维护行为

---

### 1.2 类型系统的本质

**类型是什么**：
```
类型 = 值集合 + 操作集合 + 不变量
```

**类型的作用**：
- **约束**：限定值的范围和行为
- **区分**：区分不同种类的事物
- **安全**：编译期检查，快速失败

**类型退化链条**：
```
强类型 → 弱类型
枚举   → 字符串常量 → 魔法字符串 → 硬编码
编译检查 → 运行时判断 → 运行时错误 → 不可预测
```

**类型退化信号**：
- 值被用来判断行为 → 类型信息丢失
- 编译期错误变成运行时错误 → 类型安全丧失
- 枚举退化成字符串常量 → 类型定义无效

**示例**：

```java
// 类型清晰（好）
IndexationType type = income.getIndexation();
// 编译期类型安全，IDE自动补全

// 类型退化（坏）
String type = income.getIndexation();
if ("FLOAT".equals(type)) { ... }
// 类型信息丢失，运行时判断，容易出错
```

---

### 1.3 多态的本质

**多态是什么**：
```
多态 = 同一接口 + 不同行为
```

**多态解决的问题**：
如何在不修改调用代码的情况下，扩展新的行为？

**分发机制对比**：

| 分发方式 | 时机 | 代价 | 安全性 |
|---------|------|------|--------|
| if-else | 运行时 | 分支预测失败 | 低（易遗漏） |
| switch | 运行时 | 跳转表 | 中 |
| 虚方法 | 运行时 | 间接调用 | 高 |
| 模板/泛型 | 编译期 | 零开销 | 最高 |

**多态实现方式**：
- **类型多态**：继承/接口，子类覆盖方法
- **参数多态**：泛型，类型参数化
- **策略模式**：组合 + 委托
- **访问者模式**：双分派

**本质**：用类型系统分发行为，而非运行时判断

---

### 1.4 数据结构的本质

**数据结构是什么**：
```
数据结构 = 数据组织方式 + 访问操作 + 复杂度保证
```

**核心权衡**：
- 时间 vs 空间
- 顺序访问 vs 随机访问
- 读密集 vs 写密集

**数据结构与访问模式匹配**：

| 访问模式 | 推荐结构 | 不推荐 | 原因 |
|---------|---------|--------|-----|
| 顺序遍历 | List/Array | Map | 简单高效 |
| Key查找 | Map/HashMap | List | O(1) vs O(n) |
| 范围查询 | TreeMap/排序数组 | HashMap | 有序性 |
| 去重 | Set | List | 自动去重 |

**数据模型退化**：
```
正确模型 → 模糊 → 错误模型
    ↓                 ↓
 匹配访问          不匹配
   高效             低效
```

**示例**：

```java
// 错误：用List存储，却按Key查找
List<Item> items = ...;
for (Item item : items) {
    if (item.getId() == targetId) { ... }  // O(n)
}

// 正确：用Map存储，O(1)查找
Map<Integer, Item> itemMap = items.stream()
    .collect(Collectors.toMap(Item::getId, Function.identity()));
itemMap.get(targetId);  // O(1)
```

## 二、方法论体系

### 2.1 问题识别框架

**信号 → 症状 → 病因**

```
代码异味 → 症状表现(What) → 根因分析(Why) → 设计退化(本质)
```

**识别方法**：
1. **静态扫描**：正则匹配模式
2. **指标量化**：圈复杂度、重复率
3. **人工审查**：理解业务逻辑

**退化类型分类**：

| 退化类型 | 信号 | 根因 |
|---------|------|-----|
| 类型退化 | 值判断行为 | 类型信息丢失 |
| 数据退化 | 嵌套查找 | 结构不匹配 |
| 行为退化 | 重复代码 | 缺少抽象 |
| 关系退化 | 紧耦合 | 依赖方向错误 |

---

### 2.2 问题分析框架

**三问法**：

```
Q1: 为什么会出现这个问题？ → 根因
Q2: 这个问题的本质是什么？ → 抽象
Q3: 正确的设计应该是什么？ → 目标
```

**示例分析**：

```
问题：代码中有大量if-else判断

Q1: 为什么出现？
A1: 类型信息被退化成字符串，运行时才能确定行为

Q2: 本质是什么？
A2: 类型退化，值判断代替类型区分

Q3: 正确设计是什么？
A3: 恢复类型系统，用多态分发替代运行时判断
```

**退化程度评估**：

| 退化维度 | 表现 | 本质 | 恢复方案 |
|---------|------|-----|---------|
| 类型退化 | 值判断行为 | 类型信息丢失 | 多态/策略模式 |
| 数据退化 | 嵌套查找 | 结构不匹配 | 重建索引/换容器 |
| 行为退化 | 重复代码 | 缺少抽象 | 提取抽象 |
| 关系退化 | 紧耦合 | 依赖方向错误 | 解耦/依赖注入 |

---

### 2.3 决策框架

**方案选择矩阵**：

| 问题本质 | 推荐方案 | 适用条件 | 收益 |
|---------|---------|---------|-----|
| 类型退化 | 多态/策略模式 | 行为差异明显 | 类型安全、易扩展 |
| 数据退化 | 重建索引/换容器 | 访问模式明确 | 性能提升 |
| 行为退化 | 提取抽象 | 重复代码多 | 减少重复 |
| 关系退化 | 解耦/依赖注入 | 模块边界清晰 | 降低耦合 |

**决策原则**：
1. 优先解决根本原因，而非表面症状
2. 选择收益/成本比最高的方案
3. 小步快跑，持续验证

## 四、策略模式重构标准流程

### 4.1 五步法

```
1. 识别变体
   - 找到所有if-else分支
   - 提取变化维度
   - 统计出现频次

2. 定义接口
   - 抽象公共行为
   - 定义supports()方法
   - 提供default实现

3. 实现策略
   - 每个分支一个策略类
   - 添加@Component注解
   - 实现supports()判断

4. 注入策略
   - Service注入List<Strategy>
   - 构建查找方法

5. 替换判断
   - 删除if-else
   - 多态调用strategy.method()
```

### 4.2 策略接口设计规范

```java
public interface XxxStrategy {
    
    /**
     * 判断是否支持该类型
     */
    boolean supports(String type);
    
    /**
     * 核心业务方法（可选）
     */
    default void execute(Context ctx) {
        // 默认实现或空
    }
    
    /**
     * 查询方法（可选）
     */
    default boolean isXxx() {
        return false;
    }
}
```

### 4.3 策略查找优化

```java
// ❌ 避免：每次遍历
private Strategy getStrategy(String type) {
    return strategies.stream()
        .filter(s -> s.supports(type))
        .findFirst()
        .orElseThrow();
}

// ✅ 推荐：启动时构建Map
private Map<String, Strategy> strategyMap;

@PostConstruct
public void init() {
    strategyMap = strategies.stream()
        .collect(Collectors.toMap(
            Strategy::getType,
            Function.identity()
        ));
}

public Strategy getStrategy(String type) {
    return strategyMap.get(type);  // O(1)
}
```

## 五、性能优化技术

### 5.1 算法优化矩阵

| 问题 | 原方案 | 优化方案 | 复杂度变化 |
|-----|--------|---------|-----------|
| 嵌套filter | O(n²)~O(n³) | 预建Map索引 | → O(n) |
| 不必要排序 | O(n log n) | 删除 | → O(0) |
| 频繁查找 | List遍历O(n) | Map查找 | → O(1) |

### 5.2 内存优化技术

```java
// ❌ 错误：预初始化浪费内存
private List<Item> items = new ArrayList<>();

// ✅ 正确：延迟初始化
private List<Item> items;
public List<Item> getItems() {
    if (items == null) {
        items = new ArrayList<>();
    }
    return items;
}

// ✅ 更好：指定初始容量
List<Item> list = new ArrayList<>(expectedSize);
Map<K, V> map = new HashMap<>(expectedSize * 2);
```

### 5.3 容器选择指南

| 场景 | 推荐容器 | 原因 |
|-----|---------|-----|
| 去重 | Set | 自动去重 |
| 查找 | Map | O(1)查找 |
| 顺序遍历 | List | 简单高效 |
| 不变集合 | `Collections.emptyList()` | 零内存 |

## 七、应用案例

### 7.1 类型退化修复案例

**问题识别**：
```
信号：if (IndexationType.FLOAT.getValue().equals(type)) 出现12次
位置：BondsServiceImpl, BondInstCflServiceImpl 等7个Service
模式：运行时字符串比较
```

**三问分析**：
```
Q1: 为什么出现？
A1: BondsIncome.indexation字段是String类型，类型信息丢失

Q2: 本质是什么？
A2: 类型退化，枚举退化成字符串常量

Q3: 正确设计？
A3: 
  1. 字段改为枚举类型
  2. 用策略模式分发行为
```

**解决方案**：
1. 创建IndexationTypeStrategy接口
2. 实现4个策略类：Fixed/Float/Segmented/Discount
3. Service注入策略列表
4. 删除所有if-else判断

**效果评估**：
| 指标 | 优化前 | 优化后 |
|-----|--------|--------|
| 运行时判断 | 12次 | 0次 |
| 类型安全 | 无 | 编译期检查 |
| 可扩展性 | 差 | 优 |

### 7.2 数据退化修复案例

**问题识别**：
```
信号：嵌套Stream filter，三层循环
位置：FixingServiceImpl
模式：O(n³)复杂度
```

**三问分析**：
```
Q1: 为什么出现？
A1: 数据用List存储，却需要按ID关联查找

Q2: 本质是什么？
A2: 数据结构与访问模式不匹配

Q3: 正确设计？
A3: 预构建Map索引，O(1)查找
```

**解决方案**：
```java
// 构建索引
Map<Integer, Cashflow> cashflowMap = cashflows.stream()
    .collect(Collectors.toMap(Cashflow::getId, Function.identity()));

Map<Integer, List<Reset>> resetMap = resets.stream()
    .collect(Collectors.groupingBy(Reset::getCashflowId));

// O(1)查找
Cashflow cf = cashflowMap.get(cashflowId);
List<Reset> resetList = resetMap.get(cashflowId);
```

**效果评估**：
| 指标 | 优化前 | 优化后 |
|-----|--------|--------|
| 时间复杂度 | O(n³) | O(n) |
| 性能提升 | - | 50倍 |

## 八、避坑指南

### 8.1 常见错误

| 错误 | 表现 | 解决 |
|-----|------|-----|
| 策略遗漏 | 某分支无匹配策略 | 添加DefaultStrategy兜底 |
| supports错误 | 策略匹配失败 | 单元测试验证 |
| 策略冲突 | 多个策略匹配 | 用@Order控制优先级 |
| 忘记注入 | NullPointerException | 添加@Autowired |

### 8.2 继承vs组合

```
❌ 错误继承：语义错误、内存浪费
class Child extends Parent {
    // 继承了不需要的字段和方法
}

✅ 正确组合：语义清晰、解耦
class Child {
    private Parent parent;  // 组合而非继承
}
```

### 8.3 过度设计

```
不是所有if-else都需要策略模式：
- 只出现1-2次：保持简单
- 行为无差异：不需要策略
- 稳定不变：直接用枚举switch
```
