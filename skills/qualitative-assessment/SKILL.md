# 系统定性评估技能

**核心原则**：客观、如实、深刻、尖锐，300-500字总结

## 评估目的

基于 `/full-analysis` 前四步的客观分析结果，进行定性总结和系统定位，回答：

1. **这是什么系统？**（领域定位）
2. **系统设计倾向是什么？**（面向人 vs 面向机器）
3. **开发模式是什么？**（可扩展 vs 拷贝再开发）
4. **运维模式是什么？**（自动化 vs 全人工）
5. **非功能属性如何？**（健壮性、可维护性、性能）
6. **这样的设计是否匹配其领域要求？**

## 输入来源

收集前四步技能的输出文件：

1. `/code-deconstruct` → 设计文档、ER图、架构图
2. `/code-review` → 代码质量、安全、性能问题
3. `/code-detect-dup` → 重复度分析
4. `/code-detect-problem` → 项目层面问题、重构方案

**文件查找模式**：

```bash
# 查找最近的分析输出文件
find docs/ -name "*.md" -type f | grep -E "(deconstruct|review|dup|problem)" | sort -r | head -10
```

## 评估维度

### 1. 设计维度：面向人 vs 面向机器

| 特征 | 面向人 | 面向机器 |
| ------ | ------ | ------ |
| **交互方式** | GUI/CLI交互 | API/消息队列 |
| **错误处理** | 友好提示 | 错误码/日志 |
| **配置管理** | 配置文件/界面 | 环境变量/启动参数 |
| **日志输出** | 人类可读 | 结构化/机器可解析 |

**判断标准**：

- 面向人：大量UI代码、交互逻辑、用户引导
- 面向机器：API接口、协议处理、后台服务

### 2. 开发维度：可扩展 vs 拷贝再开发

| 特征 | 可扩展设计 | 拷贝再开发 |
| ------ | ------ | ------ |
| **架构模式** | 模块化、插件化 | 单体、紧耦合 |
| **代码复用** | 抽象接口、继承 | 复制粘贴 |
| **配置方式** | 配置驱动 | 硬编码 |
| **依赖管理** | 依赖注入 | 直接实例化 |

**判断标准**：

- 可扩展：清晰的接口、配置化、插件机制
- 拷贝再开发：重复代码、相似功能不同实现

### 3. 运维维度：自动化 vs 全人工

| 特征 | 自动化运维 | 全人工运维 |
| ------ | ------ | ------ |
| **部署** | CI/CD流水线 | 手动拷贝文件 |
| **监控** | 指标收集、告警 | 人工检查日志 |
| **故障恢复** | 自动重启/切换 | 人工干预 |
| **配置更新** | 配置中心推送 | 手动修改文件 |

**判断标准**：

- 自动化：部署脚本、监控配置、健康检查
- 全人工：缺乏运维文档、手动操作步骤

### 4. 非功能属性评估

| 属性 | 评估要点 | 标准 |
| ------ | ------ | ------ |
| **健壮性** | 错误处理、异常恢复、边界条件 | 有无防御性编程、降级策略 |
| **可维护性** | 代码结构、文档、测试覆盖 | 模块清晰度、注释质量、测试完整性 |
| **性能** | 响应时间、资源使用、扩展性 | 算法复杂度、缓存策略、并发处理 |

### 5. 领域匹配度评估

| 系统类型 | 允许的宽松度 | 必须严格的要求 |
| ------ | ------ | ------ |
| **纯小工具** | 面向人、拷贝再开发、全人工 | 功能正确即可 |
| **管理类系统**<br>(ERP/OA/录单) | 弱化性能、简化设计 | 数据一致性、业务流程正确 |
| **普通报价系统**<br>(外贸/电商报价) | 中等 | 计算准确性、业务逻辑正确 |
| **金融报价系统**<br>(股票/期货/外汇报价) | 极低 | 高性能、零差错、强实时性、强一致性 |
| **量化交易系统** | 极低 | 高性能、零差错、强健壮性、风控完备 |
| **基础设施**<br>(数据库/中间件) | 极低 | 高可用、强一致、可扩展 |

**领域识别线索**：

- 业务术语：订单、客户、库存 → 业务系统
- 金融术语：报价、交易、风控 → 金融系统（注意：金融报价等同于量化交易，要求极严格）
- 技术术语：连接池、缓存、队列 → 基础设施
- 工具术语：转换、处理、分析 → 工具类

**报价系统特别说明**：

- **金融报价系统**：股票/期货/外汇报价，等同于量化交易，要求高性能、零差错、强实时性
- **普通报价系统**：外贸/电商报价，属于业务系统，中等严格度即可

## 评估流程

### 步骤1：收集分析结果

```bash
# 定位最新分析文件
timestamp=$(date +%Y%m%d)
review_file=$(find docs/review -name "code-review-*.md" | sort -r | head -1)
deconstruct_file=$(find docs/deconstruct -name "deconstruct-*.md" | sort -r | head -1)
dup_file=$(find docs/dup -name "dup-*.md" | sort -r | head -1)
problem_file=$(find docs/problem -name "problem-*.md" | sort -r | head -1)

# 读取关键信息
extract_summary() {
    echo "## 从 $1 提取的要点："
    grep -A5 -B5 "总结\|建议\|问题" "$1" | head -20
}
```

### 步骤2：维度分析

对每个维度进行打分（1-5分）：

**设计维度评分**：

- 5分：完全面向机器，结构化API，完善的错误码
- 4分：主要面向机器，但有人类可读的辅助
- 3分：混合设计，既有API也有界面
- 2分：主要面向人，但提供基础API
- 1分：完全面向人，无机器接口

**开发维度评分**：

- 5分：高度可扩展，完整插件体系
- 4分：模块化设计，良好抽象
- 3分：部分模块化，部分硬编码
- 2分：主要硬编码，少量复用
- 1分：完全拷贝再开发，高度重复

**运维维度评分**：

- 5分：全自动化，完整CI/CD/监控
- 4分：主要自动化，少量人工
- 3分：半自动化，关键步骤自动化
- 2分：主要人工，基础脚本辅助
- 1分：全人工，无自动化

**非功能属性评分**：

- 健壮性：错误处理完整性
- 可维护性：代码结构清晰度
- 性能：算法和资源使用优化度

### 步骤3：领域识别和匹配度判断

**识别步骤**：

1. 提取项目描述中的关键词
2. 分析主要业务逻辑
3. 判断系统核心价值
4. 匹配到领域分类

**匹配度判断**：

- 匹配：系统设计与领域要求一致
- 部分匹配：某些维度不符合但可接受
- 不匹配：关键维度严重偏离领域要求

### 步骤4：生成尖锐总结

**总结结构**：

1. **一句话定性**：这是什么系统，核心价值是什么
2. **设计倾向**：面向人/机器，程度如何
3. **开发模式**：可扩展性如何，是否适合长期维护
4. **运维现状**：自动化程度，运维负担
5. **非功能短板**：最突出的问题领域
6. **领域匹配度**：设计与领域要求是否匹配
7. **风险提示**：最大的潜在风险
8. **改造建议**：如果需要改进，最关键的一步是什么

**语言风格**：

- 客观：基于事实，不夸大不贬低
- 如实：真实反映现状，不回避问题
- 深刻：看到本质，不止于表面
- 尖锐：直指要害，不留情面但建设性

## 验证标准

1. **客观性**：所有结论有数据支持
2. **深刻性**：不止于表面现象，看到本质
3. **尖锐性**：不回避问题，直指要害
4. **建设性**：批评的同时给出改进方向
5. **简洁性**：总结在300-500字内

---
**核心价值**：

1. **定性定位**：从客观分析到定性判断
2. **维度全面**：设计、开发、运维、非功能多维度
3. **领域匹配**：结合系统类型评估设计合理性
4. **尖锐深刻**：提供有价值的批判性见解
5. **行动指导**：指出最关键改进方向
